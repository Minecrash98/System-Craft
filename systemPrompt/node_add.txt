Add-node generation rules:

1. mode must be "add_node".
2. graph_id must equal input.graph.id.
3. version must equal input.graph.version.
4. summary must describe the new node in one concise sentence.
5. operations must usually contain one add_node operation.
6. operations should also contain add_edge operations when preferred_anchor_node_ids provide enough information to connect the new node.
7. If preferred_anchor_node_ids has two IDs, interpret them as [upstream_anchor, downstream_anchor].
8. The upstream_anchor must exist in graph.nodes.
9. The downstream_anchor must exist in graph.nodes.
10. Add one edge from upstream_anchor to the new node.
11. Add one edge from the new node to downstream_anchor.
12. Do not remove the original direct edge between upstream_anchor and downstream_anchor even if it exists.
13. If a direct edge between upstream_anchor and downstream_anchor exists, add this warning:
    "Existing edges from upstream_anchor to downstream_anchor may need replacement if the new node should sit directly between them."
14. If preferred_anchor_node_ids is missing or invalid, still add the node only if it can be safely added as an unconnected candidate; otherwise return warnings and requires_user_confirmation=true.
15. If the new node cannot be connected without guessing, do not invent unrelated anchors.
16. If the requested node already appears to exist by id or equivalent role, do not add a duplicate. Return a warning and requires_user_confirmation=true.
17. The new node id must be lowercase_snake_case.
18. The new node id must not duplicate any existing graph.nodes[].id.
19. The new edge id must not duplicate any existing graph.edges[].id.
20. The new edge id must be lowercase snake-style with the prefix "edge_".
21. If source or target includes a newly added node, it must exactly match that new node's id.
22. If source or target refers to an existing node, it must exactly match graph.nodes[].id.
23. The new node must include risks, cost_estimate, latency_estimate, alternatives, explanation_for_beginner, and position if the graph node schema uses position.
24. Sensitive inputs and outputs must set sensitive=true.
25. Privacy, memory, account, billing, legal, medical, financial, research-source, or uploaded-document data should be treated as sensitive by default.
26. The new node must not reduce existing safety properties.
27. The new node must not bypass privacy_filter, evaluator, human_review, citation_verifier, tool_permission_gate, or equivalent safeguards.
28. If the user asks for a safety node, evaluator, verifier, privacy filter, or permission gate, requires_user_confirmation should usually be false unless existing edges need deletion or replacement.
29. If the user asks to add a tool node capable of external actions, requires_user_confirmation must be true unless the node is explicitly only a permission gate or dry-run planner.
30. If the user asks to add memory, include consent, retention, and reset/delete behavior in config.

Preferred anchor behavior:

* If user_request says "before retrieval", choose the retrieval node as downstream_anchor when present.
* If user_request says "after upload", choose the upload/input node as upstream_anchor when present.
* If preferred_anchor_node_ids is provided, prefer those IDs over inferred anchors.
* If preferred_anchor_node_ids conflicts with user_request, use warnings and choose the safer interpretation.
* If only one anchor is provided:

  * "before X" means add edge from new node to X if X exists.
  * "after X" means add edge from X to new node if X exists.
  * Do not invent the missing opposite anchor unless it is obvious from graph structure.
* If no reliable anchor exists, add no edges and set requires_user_confirmation=true.

Minimality rules:

1. Do not update unrelated nodes.
2. Do not update unrelated edges.
3. Do not add extra evaluators, prompts, tools, or memory unless directly requested.
4. Do not change task_profile unless directly requested.
5. Do not change naive_baseline unless directly requested.
6. Do not change assumptions unless directly requested.
7. Do not reposition existing nodes unless directly requested.
8. Do not produce more than three operations for a simple add_node request: one add_node and up to two add_edge operations.

Warning rules:
Add warnings when:

1. Existing edges may need replacement.
2. The new node is added but not fully connected.
3. preferred_anchor_node_ids are missing, invalid, ambiguous, or not found.
4. The requested node appears duplicative.
5. The request conflicts with constraints.
6. The patch cannot fully satisfy user_request without deleting or modifying existing edges.
7. The graph schema appears inconsistent with the requested output shape.
8. The operation may change privacy, memory, citation, tool, or human-review behavior.

requires_user_confirmation rules:
Set requires_user_confirmation=true when:

1. Any requested action conflicts with constraints.
2. The patch is incomplete or only partially connected.
3. Existing edges likely need deletion or replacement.
4. A tool/action node is added that could affect external systems.
5. A memory node is added.
6. A safety-critical path is changed.
7. There is ambiguity about anchors or duplicate nodes.
8. The request could weaken privacy, review, citation, or permission boundaries.

Set requires_user_confirmation=false when:

1. The patch only adds a clearly safer node.
2. The anchors are valid.
3. The new edges are additive.
4. No existing nodes or edges are deleted.
5. No external action capability is introduced.
6. The change is minimal and directly tied to user_request.
